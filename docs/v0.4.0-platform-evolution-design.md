# spec-superflow v0.4.0 Platform Evolution — Design Document

**Date:** 2026-06-29
**Version:** 0.3.0 → 0.4.0
**Goal:** Transform spec-superflow from a pure-skill workflow into a configurable, CLI-operable platform while maintaining zero npm dependencies.

---

## 1. Overview

v0.4.0 evolves spec-superflow across 7 dimensions:

| # | Dimension | Impact Layer |
|---|-----------|-------------|
| 1 | CLI Toolchain | New `scripts/spec-superflow.mjs` + bin |
| 2 | Configurable Schema (lightweight) | New `spec-superflow.config.json` + config loader |
| 3 | Multi-language Tokenizer | New `src/validation/tokenizer.ts` |
| 4 | git worktree Isolation | SKILL.md guidance (execution-governor) |
| 5 | spec-syncer Conflict Detection | New `Validator.detectSyncConflicts()` |
| 6 | Version Sync Automation | CLI `ssf version` command |
| 7 | Documentation Sync | CLI `ssf doctor` extension |

---

## 3. Layer 1: Configuration System

### 3.1 File: `spec-superflow.config.json`

Optional. When absent, all settings use built-in defaults (identical to v0.3.0 behavior).

```json
{
  "artifacts": {
    "order": ["proposal", "specs", "design", "tasks", "execution-contract"],
    "skip": []
  },
  "execution": {
    "inlineThreshold": 3,
    "abandonmentReasonMinLength": 50,
    "defaultLanguage": "auto"
  },
  "verification": {
    "language": "auto"
  }
}
```

### 3.2 Config Fields

| Field | Type | Default | Purpose |
|-------|------|---------|---------|
| `artifacts.order` | string[] | 5-item default | Artifact generation sequence |
| `artifacts.skip` | string[] | `[]` | Artifacts to skip entirely |
| `execution.inlineThreshold` | number | `3` | Tasks ≤ N → inline mode |
| `execution.abandonmentReasonMinLength` | number | `50` | Min chars for abandonment reason |
| `execution.defaultLanguage` | string | `"auto"` | Default content language |
| `verification.language` | string | `"auto"` | Tokenizer language for verification |

### 3.3 Config Loading

New module: `scripts/lib/config-loader.mjs`

```
loadConfig(projectRoot) → merged config object
  1. Read spec-superflow.config.json from projectRoot (or walk up to find it)
  2. Deep merge with built-in defaults
  3. Validate field types
  4. Return merged config
```

CLI commands and SKILL.md scripts both call `loadConfig()`. SKILL.md references config via:
```
bash scripts/get-config.sh <field-path>
```

New helper script: `scripts/get-config` (bash) — reads JSON config and outputs a specific field value. Skills use this to query current settings at runtime.

### 3.4 How Skills Consume Config

Each SKILL.md that references a configurable value is updated to include a config check step:

```markdown
### Config Check
Before proceeding, check the project configuration:
- Run: `bash "${CLAUDE_PLUGIN_ROOT}/scripts/get-config" execution.inlineThreshold`
- If the script returns a value, use it; otherwise use the default (3)
```

This keeps skills config-aware without requiring a runtime engine.

## 4. Layer 2: CLI Toolchain

### 4.1 Entry Point

File: `scripts/spec-superflow.mjs` (ESM, `#!/usr/bin/env node`)

Uses `node:util.parseArgs` for argument parsing. Zero external dependencies.

```json
// package.json additions
{
  "bin": {
    "ssf": "./scripts/spec-superflow.mjs",
    "spec-superflow": "./scripts/spec-superflow.mjs"
  }
}
```

### 4.2 Command: `ssf list`

Lists all change directories under `changes/` with status.

**Output format:**
```
Changes:
  v0.3.0-workflow-enhancements  [CLOSED]  5 specs synced
  v0.4.0-platform-evolution     [ACTIVE]  4/7 artifacts ready
```

**Implementation:** `scripts/lib/cmd-list.mjs`
- Scan `changes/` for directories containing `proposal.md`
- Check for `execution-contract.md` (exists → bridged or executing)
- Check for `abandonment-summary.md` (exists → abandoned)
- Check specs/ sync status by comparing delta spec names against `specs/` root

### 4.3 Command: `ssf validate <dir>`

Validates artifacts in a change directory.

**Checks:**
- proposal.md: `validateChangeContent()`
- Each `specs/*/spec.md`: `validateSpecContent()`
- Each delta spec: `validateDeltaSpec()`
- Cross-artifact consistency (proposal scope vs spec capabilities)

**Exit codes:** 0 = all pass, 1 = validation failures, 2 = file not found

**Implementation:** `scripts/lib/cmd-validate.mjs` — imports `Validator` from `dist/index.js`

### 4.4 Command: `ssf doctor`

Health check for the spec-superflow installation and project.

**Checks:**
1. Version consistency across all manifest files
2. hooks.json format validity
3. skills/ directory completeness (expected skill count)
4. dist/ compilation artifacts exist
5. spec-superflow.config.json validity (if present)
6. Node.js version ≥ 22
7. README/docs consistency with code (see Section 7.4)

**Output format:**
```
spec-superflow doctor:
  ✅ Version: 0.4.0 (consistent across 5 manifests)
  ✅ Hooks: valid format
  ✅ Skills: 9/9 present
  ✅ dist/: compiled
  ⚠️  README: "Current Status" shows v0.3.0, package.json shows v0.4.0
  ✅ Node.js: v22.4.0
```

**Implementation:** `scripts/lib/cmd-doctor.mjs`

### 4.5 Command: `ssf version <semver>`

One-command version synchronization across all manifest files.

**Targets:**
1. `package.json` → `version`
2. `.claude-plugin/plugin.json` → `version`
3. `.claude-plugin/marketplace.json` → `plugins[0].version`
4. `.cursor-plugin/plugin.json` → `version`
5. `gemini-extension.json` → `version`

**Behavior:**
- Reads current versions from all files
- Shows diff: `package.json: 0.3.0 → 0.4.0`
- Writes new version to all files
- Outputs sync report
- Does NOT auto-commit

**Implementation:** `scripts/lib/cmd-version.mjs`

### 4.6 Command: `ssf sync <change-dir>`

Merges delta specs from a change into the main `specs/` directory.

**Behavior:**
1. Load delta specs from `<change-dir>/specs/`
2. Call `detectSyncConflicts()` to check for conflicts with other unsynced changes
3. No conflicts → merge each delta spec into `specs/<capability>/spec.md`
4. Conflicts found → display conflict list, ask user to resolve order

**Implementation:** `scripts/lib/cmd-sync.mjs` — imports parsers and validator from `dist/index.js`

### 4.7 Command: `ssf config [options]`

Display or modify configuration.

**Modes:**
- No args → display current effective config (defaults + overrides merged)
- `--get <path>` → output one config field
- `--set <path>=<value>` → update config file

**Implementation:** `scripts/lib/cmd-config.mjs` — uses `config-loader.mjs`

## 5. Layer 3: Engine Extensions

### 5.1 Multi-language Tokenizer

**New file:** `src/validation/tokenizer.ts`

**Exports:**
```typescript
export function tokenize(text: string, language?: 'auto' | 'en' | 'zh'): Set<string>;
export function detectLanguage(text: string): 'en' | 'zh' | 'mixed';
```

**English tokenizer** (extracted from current `stem()` method):
1. Lowercase
2. Split on non-alphanumeric characters
3. Remove stop words (the, a, an, is, are, was, were, be, been, being, have, has, had, do, does, did, will, would, shall, should, may, might, can, could, of, in, to, for, with, on, at, by, from, as, into, through, during, before, after, above, below, between, out, off, over, under, again, further, then, once)
4. Apply stemming rules (existing suffix stripping)
5. Filter tokens with length ≤ 3
6. Return Set<string>

**Chinese tokenizer** (new):
1. Split on Chinese punctuation: `，。！？；：、""''（）【】《》`
2. Extract continuous CJK character sequences (Unicode ranges: U+4E00-U+9FFF, U+3400-U+4DBF, U+20000-U+2A6DF)
3. Also extract ASCII words from mixed content
4. Remove Chinese stop words (的, 了, 在, 是, 我, 有, 和, 就, 不, 人, 都, 一, 一个, 上, 也, 很, 到, 说, 要, 去, 你, 会, 着, 没有, 看, 好, 自己, 这)
5. Keep tokens with length ≥ 2
6. Return Set<string>

**Auto-detection** (`detectLanguage()`):
1. Count CJK characters in text
2. CJK ratio > 30% → `'zh'`
3. CJK ratio ≤ 30% and has ASCII words → `'en'`
4. Both significant → `'mixed'`

**Mixed mode** (when `detectLanguage()` returns `'mixed'`):
- Run both tokenizers
- Union the resulting token sets

### 5.2 Integration with validateImplementation()

Modify `Validator.validateImplementation()`:
1. Replace inline `stem()` calls with `tokenize(diffSummary, config?.verification?.language)`
2. Replace inline `stem()` calls with `tokenize(requirementName, config?.verification?.language)`
3. Add optional `config` parameter: `validateImplementation(diffSummary, specContent, designContent, config?: { verification?: { language?: string } })`
4. Completeness dimension: both requirement name and diff summary go through `tokenize()`
5. Coherence dimension: decision names and diff summary go through `tokenize()`

### 5.3 spec-syncer Conflict Detection

**New method on Validator:**

```typescript
detectSyncConflicts(deltaSpecs: Array<{ changeName: string, content: string }>): ConflictReport;

interface ConflictReport {
  hasConflicts: boolean;
  conflicts: Array<{
    requirement: string;
    spec: string;
    changes: string[];  // change names that both modify this requirement
  }>;
}
```

**Algorithm:**
1. For each delta spec, parse MODIFIED/RENAMED requirement names
2. Build a map: `requirementName → [changeName, ...]`
3. Any requirement with ≥ 2 changes → conflict entry
4. Return `ConflictReport`

### 5.4 git worktree Isolation

**Pure SKILL.md guidance** — no code changes.

Add to `execution-governor/SKILL.md`:

```markdown
### Worktree Isolation (Optional, Recommended)

Before starting execution, check the current branch:

1. Run: `git branch --show-current`
2. If on main/master branch:
   - Create worktree: `git worktree add ../<project>-<change-name> -b <change-name>`
   - Change to worktree directory
   - Execute all tasks in the worktree
3. If already on a feature branch → proceed normally
4. After completion, remind the user:
   - "Worktree ready for merge. Run: git merge <change-name> && git worktree remove <path>"

If worktree isolation cannot be created on `main`/`master` → stop and ask the user for explicit approval before editing the current checkout.
```

### 5.5 Documentation Sync (doctor extension)

**New checks in `cmd-doctor.mjs`:**

| Check | How |
|-------|-----|
| README version matches package.json | Regex extract version from README "Current Status" section |
| CHANGELOG has current version entry | Search for `## [<version>]` heading |

**Output:** Report inconsistencies as warnings (not errors). Do not auto-fix.

## 7. Risks and Trade-offs

### R1: Scripts in ESM vs CJS

**Risk:** `dist/` is compiled as CommonJS (tsconfig target: NodeNext). CLI scripts are `.mjs` (ESM).

**Mitigation:** CLI scripts import from `dist/index.js` using dynamic `import()`. Test this interop in the test suite.

### R2: node:util.parseArgs Limitations

**Risk:** `parseArgs` is less feature-rich than commander/yargs. No auto-generated help text, no subcommand nesting.

**Mitigation:** Hand-write help text for each subcommand. Use `--help` flag detection in each cmd module. Acceptable trade-off for zero dependencies.

### R3: Chinese Tokenizer Without Segmentation Library

**Risk:** Character-level CJK splitting misses compound words (e.g., "令牌桶" should be one token, not three characters).

**Mitigation:** Use 2-character sliding window as additional tokens. "令牌桶" → tokens: ["令牌", "牌桶", "令牌桶"]. This over-generates but avoids false negatives. Acceptable for verification (we want to catch missing requirements, not produce perfect segmentation).

### R4: Config File Discovery

**Risk:** Walking up directories to find config may pick up wrong file in monorepos.

**Mitigation:** Config lookup order: (1) current directory, (2) git root (via `git rev-parse --show-toplevel`), (3) home directory. First match wins. Document this behavior.

### R5: Breaking Changes to validateImplementation() Signature

**Risk:** Adding optional `config` parameter changes the method signature.

**Mitigation:** New parameter is optional with default `undefined` (equivalent to current behavior). Existing callers work unchanged.
